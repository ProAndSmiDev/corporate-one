## Особенности ветки «GH-PAGES»:

### 1. Прежде чем пушить, необходимо проверить, что на локалке отрабатывает все корректно
### 2. svg-иконки могут не отрабатывать, так как им нужен сервер, их игнорируем
### 3. Если проект клонируется, необходимо, чтобы хотя бы раз сборка была запущена в этой ветви

## Особенности при работе с веткой:
### 1. Если возникают какие-то ошибки в верстке - правим их в своей ветке от мастера
### 2. Если ошибок нет, тогда сохраняем свои изменения и пушим ветку в проект
### 3. Данная ветка может заходить далеко вперед по сравнению с мастером
### 4. Запрещается данную ветку пушить в ветку мастер
### 5. Разрешено принимать изменения в данную ветку с мастера, решая конфликты, если вдруг они возникли
### 6. Все коммиты должны начинаться с «GHPages:» в ином случае пулл-реквест отправляется на доработку

## Тонкости проекта (**ОБЯЗАТЕЛЬНО К ПРОЧТЕНИЮ**):
  ### 1. BEM-naming:
  #### 1.1. Класс для блока пишется в шашлычной нотации (kebab-case)
  #### 1.2. Класс для элемента пишется в шашлычной нотации после **двух** знаков подчеркивания (kebab-case__kebab-case)
  #### 1.3. Класс для модификатора пишется в шашлычной нотации после **двух** знаков тире (kebab-case__kebab-case--kebab-case)
  #### 1.4. Виды модификаторов:
  ##### 1.4.1. Модификатор **is** - пишется в том случае, если мы что-то модифицируем или меняем состояние компонента (kebab-case--is-size-xl, в данном случае модифицируем размер шрифта или блока)
  ##### 1.4.2. Модификатор **theme** - пишется в том случае, если мы меняем тему блока или элемента (kebab-case--theme-primary, в данном случае устанавливаем первичную тему компоненту)
  ##### 1.4.3. Модификатор **in** - пишется в том случае, если мы указываем местонахождение того или иного компонента, если данный компонент используется в нескольких местах (kebab-case--in-header, в данном случае компонент используется в шапке и нужны стили конкретно для этого места)
  ### 2. Типы классов:
  #### 2.1. Обычный класс по BEM - .menu, .first-screen
  #### 2.2. Утилитарный класс (это те классы, которые являются глобальными и не поддаются изменениям, имеющие стандартные стили для всех случаев) - .u-wrapper, .u-button
  #### 2.3. Скриптовый класс (это те классы, которые необходимы для того, чтобы накинуть логику, написанную на JS, обычно называется префиксом и тем, какое действие мы создаем, **Этот класс не для стилей!**) - .js-toggle-menu, .js-follow-to-the-top
  ### 3. Из чего состоят компоненты:
  #### 3.1. data.json - файл, который находится в папке компонента и хранит все его данные. На каждый компонент приходится по одному такому файлу
  #### 3.2. styles.scss - файл, который находится в папке компонента и хранит все его стили для корректного отображения. На каждый компонент приходится по одному такому файлу и подключаются в ***/app/assets/scss/bundle/components.scss*** в блоке Components' styles
  #### 3.3. template.pug - файл, который находится в папке компонента и хранит шаблон его верстки. На каждый компонент приходится по одному такому файлу. Каждый файл может содержать дополнительное подключение дочерних компонентов
  ### 4. Компоненты:
  #### 4.1. Каждый компонент может содержать только один стилевой (styles.scss), контентный(data.json) или шаблонный файл(template.pug)
  #### 4.2. Каждый компонент может содержать необходимое количество дочерних компонент (пример: шапка - она состоит из лого + меню + телефон)
  #### 4.3. Каждый компонент именуется в верблюжьей нотации с заглавной буквы (пример: Header, MenuLink)
  #### 4.4. Папка, в которой хранятся компоненты, называется components и должна быть в папке app
  ### 5. Папка Assets:
  #### 5.1. Папка **data** не должна поддаваться изменениям ни при каких обстоятельствах, исключение: сборка самого проекта
  #### 5.2. Папка **fonts** содержит только папки и файлы в них со шрифтами, никакие другие файлы восприниматься сборкой **НЕ БУДУТ** (пример: Arial/Arial-Regular.ttf)
  #### 5.3. Папка **img** может содержать папки и файлы с картинками в форматах .png, .jpg, .jpeg. Файлы другого формата будут успешно проигнорированы сборкой проекта. Фото в данной папке оптимизируются и передаются на прод в том формате в котором они были изначально
  #### 5.4. Папка **scss** содержит стандартные и необходимые стили для того, чтобы макет был успешно отредактирован без каких-либо лишних манипуляций со стороны верстальщика
  #### 5.5. Папка **svg** может содержать только файлы .svg формата, любые другие форматы будут успешно проигнорированы. В проде создает sprite.svg с айдишниками которые именуются по названию самого файла (пример: была иконка метки на карте с именем файла geo.svg, в спрайте будет id="geo" для обращения к этой иконке через use)
  #### 5.6 Папка **webp** может содержать папки и файлы в формате .png, .jpg, .jpeg. Любые другие форматы будут успешно проигнорированы сборкой проекта. Фото в данной папке оптимизируются, конвертируются в вебп и передаются в прод. Папка вынесена отдельно, так как некоторые фото может не понадобится переводить в формат .webp (например те фото, которые у нас задаются в background-image свойстве и в других любых свойствах, которые позволяют подключать фото). При работе с данной папкой очень важно соблюдать ту же иерархию фото, которая в папке **img** для того, чтобы фото в формате .webp было в том же месте на проде что и оригинал.
  ### 6. Библиотеки:
  #### 6.1. AOS - библиотека для работы с анимациями, работающие при скролле.
  #### 6.2. Swiper - библиотека для работы со слайдером
  #### 6.3. Leaflet - библиотека для работы с гео-картами на сайте
  ### 7. Каждый проект сделан с уже готовыми настройками для работы, которые состоят из следующих файлов:
  #### 7.1. .editorconfig - настройки для редактора кода, позволяющие удобно работать в любой программе
  #### 7.2. .eslintrc.json - настройки для JS - позволяющий работать на современном синтаксисе языка
  #### 7.3. .stylelintrc.cjs - настройки для удобного использования и написания стилей
  #### 7.4. .gitignore - настройки для git, чтобы не тянуть в проект ничего лишнего (фича для тех, кто работает с кодом)
  #### 7.5. gulpfile.js - настройки проекта, позволяющие работать полноценно с TS, JS, SCSS, PUG и другая полезная поддержка сайта при его работе или при работе с ним
  #### 7.6. LICENSE - файл, в котором расписано по какой лицензии актуален данный шаблон, без него дополнительные услуги по шаблону оказываться **не будут**
  #### 7.7. package.json - зависимости, на котором построен весь проект. Графа author не должна изменяться или удаляться, в ином случае при заказе дополнительных услуг (бесплатных или платных) **вы получаете отказ на их предоставление**
